数据没有用现成的Kaggle数据集，而是自己写Python脚本生成的——因为生成数据的过程本身就要求你理解每张表的业务含义和表之间的关系，这比直接拿现成数据学到的多。

具体做了什么

第一步：搭数据库

ecommerce_db
├── users          — 用户表（5000人）
├── categories     — 类目（10个）
├── products       — 商品（200个）
├── user_behavior  — 行为日志（5万条：浏览/加购/收藏）
├── orders         — 订单（1万笔）
├── order_items    — 订单明细（2.1万条）
└── refunds        — 退款记录（300条）

7张表，表之间用外键关联。行为表到用户表、订单表到用户表、订单明细到商品表——模拟的是真实电商的数据结构。索引加在了经常用来筛的字段上（user_id、order_date、behavior_type）。

第二步：写SQL做分析

写了11个核心查询，每个对应一个业务问题：

分析目标用的SQL技术
用户行为漏斗（浏览→加购→下单→支付）CTE + COUNT DISTINCT + CASE WHEN
RFM用户分群（6层价值分层）NTILE窗口函数 + CASE WHEN
GMV日趋势 + 7日移动平均窗口函数 AVG OVER + ROWS BETWEEN
品类销售排名 & 占比SUM + 子查询计算占比
时段热力分析HOUR() + GROUP BY + 多维度聚合
退款率分析（按品类/按城市）LEFT JOIN + 比率计算
用户留存分析自关联 + 首次行为锚定

第三步：用Python出图

pandas读数据 → matplotlib画图 → 导出8张PNG。不是什么复杂操作，但是把分析结果从"SQL跑出来的一堆数字"变成了"一眼能看懂的图表"。

第四步：搭Power BI看板

把数据导入Power BI Desktop，做了个交互式管理驾驶舱。可以按日期、类目、城市筛选，鼠标点哪就下钻到哪。这个主要是为了让分析结果能"给别人看"——不是只有自己能看懂的数字。

分析出来什么东西

漏斗分析：下单的人里34.6%最后没付钱

浏览 4,964 → 加购 4,127（83.1%）→ 下单 3,775（76.0%）→ 支付 2,467（65.4%）。下单到支付这一跳掉了最多。

RFM分群：19%的用户已经流失了

6层用户：85个"重要价值用户"人均消费2,130元，最值钱。但947人已经流失，如果能把其中一半通过优惠券拉回来，GMV能涨不少。

GMV日趋势

30天总GMV 368.7万，日均约1.97万。有明显的周末效应——周五六日比工作日高出一截。
母婴用品GMV最高，图书文娱垫底
18-21点占全天行为量的45%
整体退款率0.6%，在正常范围
一线城市贡献最大

基于数据的建议

支付转化流程要优化
下单3,775人 → 支付2,467人，中间消失了1,308人。可能是支付页面加载慢、支付方式不够（比如没有花呗/白条），或者用户下单后又去比价了。这是影响GMV最大的一个环节，优先级最高。
流失预警机制应该提前做
947个流失用户不是突然消失的——从他们的行为数据里应该能找到规律（比如连续N天没有浏览行为）。与其等用户走了再花大价钱召回，不如提前识别快要流失的用户，推送优惠券或者活动提醒。
晚高峰18-21点是黄金窗口
近一半的行为集中在这3小时。直播、秒杀、Push推送都应该安排在这个时段，ROI最高。
品类结构有优化空间
母婴用品卖得好，但图书文娱GMV很低。需要搞清楚是用户没有买书的需求，还是我们的书选得不对——这决定了是砍品类还是换选品。

做完之后的感受

  • 之前学SQL都是做练习题，一个查询一张表。这次真正感受到复杂业务问题需要把多个表连起来、嵌套好几层子查询——写的时候卡了好几次，但写完确实对SQL理解深了很多
  • 数据分析最重要的不是跑SQL和画图，而是从数字里看出业务问题。漏斗转好还是转差、哪个环节掉人最多、为什么——这个判断力的锻炼比技术本身更有价值
  • Python和BI各有所长。Python适合做深入分析（比如RFM分群的计算逻辑比较复杂），BI适合把结果做成可交互的看板，方便给不懂技术的人看
  • 数据是仿真的，但方法是真的。后来看面经里的业务分析题（"GMV下降了怎么分析""怎么给用户分层"），发现这个项目直接就能拿来当答案
